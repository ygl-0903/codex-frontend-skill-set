# skill-generalize-template 设计

## 目标

创建一个可复用 skill，把一个或多个项目专用 skill 改写成通用模版，或在明确要求下直接修改原 skill。处理范围不只限于仓库内模块 skill，也包括 `.agents`、`.claude`、`.codex` 下的 skill 目录。

## 输入与输出

- 输入支持 skill 名列表和 skill 目录路径列表，两种可以混用
- 处理前统一解析成实际目录
- skill 名默认按 `.agents/skills`、`.claude/skills`、`.codex/skills` 顺序定位
- 同名 skill 多处存在时，必须等待用户指定来源
- 默认输出为 `<source-skill-dir>-template/SKILL.md`
- 只有用户明确要求时才直接修改原 skill

## 设计选择

采用轻量规则型方案，而不是让 skill 同时产出变量 schema 或项目映射文件。

原因：

- 当前目标是稳定生成可复制的通用模版，并允许在明确授权下直接改写原 skill
- 现有 `module-*` skill 结构简单，主要差异集中在路径、栈、流程命名
- 后续还要兼容 Claude 和 Codex 两套目录，因此先把“项目绑定项”和“平台格式差异”分开
- 先把“抽离项目绑定项”的规则沉淀下来，再做下一步“项目梳理并生成项目版 skill 包”会更稳

## 核心抽象规则

保留：

- 阶段职责
- 执行步骤顺序
- 风险与停机条件
- 输出边界

抽离成占位符：

- 固定目录与文件
- 固定技术栈
- 固定参考页位置
- 固定需求文档路径
- 团队私有命名与交付路径
- 项目私有命令、入口、约定文件

删除或压缩：

- 纯示例性项目路径
- 没有复用价值的项目背景描述

## 占位符规范

- 统一使用 `<snake_case_name>`
- 先抽象“职责”，再抽象“实现”
- 同类信息使用同一命名，避免多个模版出现不同写法
- Claude/Codex 平台差异不默认抽成占位符，除非它本身就是项目变量

## 已知限制

- 当前 skill 默认只生成 `SKILL.md`，不会自动产出 schema、映射表或生成器配置
- 对严重依赖外部 references 的复杂 skill，仍需要人工复核抽象结果
- 当前仓库里没有 `.claude/skills` 与 `.codex/skills` 的完整样例，相关兼容规则基于目录约定设计
- 当前目录不是 git 仓库，无法完成设计文档提交
