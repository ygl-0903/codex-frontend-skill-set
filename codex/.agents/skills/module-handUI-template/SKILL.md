# 设计稿回填

## 适用时机

- 页面功能已经按原型或需求先开发完成，后续拿到设计稿，需要把页面样式对齐到视觉稿。
- 当前任务以视觉还原为主，允许顺带做模板层级、区块顺序、按钮位置这类结构跟随改动。
- 页面已有接口、登录态、跳转和主流程，不准备在这一轮重做业务逻辑。

## 目标与边界

- 以最小业务扰动完成设计稿回填。
- 优先复用现有数据流、事件方法和组件边界，不为了“看起来更整洁”顺手重写业务。
- 允许为了贴合设计稿调整模板结构，但这些调整不能改变页面原有业务语义。

## 先读什么

1. 如果有 `docs/modules/<feature>.md`，先读需求卡，确认这个页面原本的业务目标和状态流。
2. 读取 `AGENTS.md` 与 `bd-fe-conventions`。
3. 读取当前页面及其同目录组件，明确哪些是现有业务区块，哪些只是样式容器。
4. 如果仓库里有同域参考页，优先看视觉最接近的旧页面，确认现有项目的卡片、顶部、按钮和弹窗写法。
5. 在开始回填前，先判断设计稿中的视觉元素哪些可直接用 CSS 高质量还原，哪些纯 CSS 还原度偏低，或更适合导出图片/资源处理。

## 需要的输入

- 目标页面路径，必要时补上关联组件路径。
- 设计稿、标注稿、截图或清晰的视觉说明。
- 当前页面对应的原型、需求卡或可运行页面，用来判断哪些是视觉变更，哪些已经越界成需求变更。

## 改动边界

### 属于 `module-handUI`

- 修改颜色、字号、字重、行高、圆角、阴影、边框、背景、渐变、图标、插图和间距。
- 调整页面布局、区块顺序、卡片拆分或合并、按钮摆放位置、信息层级和留白。
- 新增包裹层、吸底栏、头图区、占位区、状态样式类名等，只为承接视觉结构。
- 调整 template 层级、局部显隐、事件挂载位置，只要事件含义和业务结果不变。
- 为配合新布局，新增少量展示态字段、布尔开关、派生文案或局部组件拆分。

### 需要谨慎判断

- 设计稿把原有入口从列表项内挪到页面顶部，若仍调用同一个方法、同一条链路，可按 handUI 处理。
- 设计稿要求新增一个视觉状态，但这个状态其实可以由现有数据直接推导，可按 handUI 处理。
- 页面结构调整后，原方法需要轻微改动才能继续工作，可以改，但目标仍然是保持原有业务语义不变。
- 设计稿要求展示某个文案或指标，但当前接口还没返回：仍然按设计稿把 UI 结构做出来，在代码里明确注释“待接口字段补齐”，并使用占位值，不要擅自改成别的展示内容。

### 不属于 `module-handUI`

- 新增、删除或重写接口调用。
- 修改请求参数、返回字段消费方式、字段语义或接口时序。
- 改页面跳转规则、提交流程、登录态、权限判断、缓存策略。
- 因设计稿变化而新增真实业务状态、审批分支、表单校验规则或新的功能入口。
- 任何已经从“视觉回填”升级成“需求变更”或“功能改造”的内容。

## 执行步骤

1. 先把设计稿与当前页面逐块对照，标出哪些是纯样式差异，哪些会引出结构调整。
2. 逐块判断视觉元素的 CSS 可还原性，把纯 CSS 难以高质量还原的元素整理到 `docs/modules/<feature>.md`，至少包含元素名称、所在区域、低还原度原因和建议方案；如果需求卡不存在，先按模块补建。
3. 先判断结构调整是否会改变业务语义：
   - 不会改变：继续按 handUI 处理。
   - 会改变：停下来标记为超出范围，提示用户拆到功能改造阶段。
3. 实施时优先保留现有 `<event_methods_name-methods>`、接口调用、`<lifecycle_pattern-生命周期>` 和主要数据对象，只改模板结构、样式和事件落点。
   - 如果设计稿里的展示字段暂无接口来源，优先保留设计稿对应的标签和位置，在代码中写清楚待补字段，不要因为“目前没数据”就换成别的字段或删除该区块。
4. 页面明显变复杂时，把纯展示区块拆到同目录组件，不要把所有视觉改动继续堆进单个 `<single_file_component_ext-.vue>`。
5. 资源优先复用 `<static_assets_dir-static/>`、`<public_path_constant-APP_PUBLIC_PATH>` 和现有 `<cross_platform_component_library-antd 组件>`，不额外引入不必要依赖。
6. 改完后回看页面的 loading、empty、弹窗、安全区、滚动区和吸底按钮，避免视觉改完把交互层级打乱。

## 任务特有提醒

- 页面结构、`className` 命名和 CSS 组织细则统一以 `../bd-fe-conventions/references/page-structure-and-css.md` 为准，不在当前 skill 重复维护一整套。
- 回填时允许新增容器，但新增容器必须有明确职责：布局层、视觉承接层或业务块层，不能只是为了多套一层。
- 同一页面如果要顺手修正历史命名，只改当前正在处理的区块，避免升级成整页重构。
- 如果一个区块已经有独立标题、背景、状态或后续拆组件价值，可以在本轮升级成独立 block；但不要借机重写整页结构。

## 输出要求

- 优先给出改动页面和组件路径，不贴大段源码。
- 明确说明这轮是：
  - 纯视觉回填
  - 视觉回填 + 结构跟随调整
  - 已超出 handUI，需升级到功能改造
- 明确说明哪些视觉元素已判断为不适合纯 CSS 高质量还原，并确认这些内容已写入 `docs/modules/<feature>.md`。
- 如果有超出范围的点，逐条列出，并说明为什么它已经不是单纯 UI 改动。

## 收尾建议

- 至少自查页面在 Web 管理后台场景下能正常渲染，没有因为结构调整导致点击区域、滚动容器或弹窗层级异常。
- 如果只是视觉稿回填完成，下一轮通常进入 `module-polish`。
- 如果设计稿实质上引入了新流程、新状态或新接口，下一轮不要继续套 `module-handUI`，应转入正常模块开发流程。
