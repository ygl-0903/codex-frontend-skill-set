# 页面与组件优化

## 适用时机

- 页面或组件已经可用，但代码偏乱、状态分散、兼容分支过多。
- 需求明确为“优化页面代码结构 / 展示逻辑”，而不是重做功能。
- 用户能提供明确页面路径或组件路径。
- 本次目标是整理现有实现，不改变核心功能、接口协议和页面语义。

## 不适用时机

- 需求本质是新增功能、重做交互、改视觉方案或换接口。
- 目标文件逻辑强依赖多个公共模块，且无法安全判断哪些代码可删。
- 需要改动公共目录、公共配置、共享组件或多个页面共用流程，才能完成优化。
- 后端字段定义本身不清晰，继续删兼容会直接带来联调风险。

## 核心目标

1. 不影响现有功能。这是第一优先级。
2. 简化流程代码，减少绕来绕去的中间层。
3. 删除不必要的兼容逻辑，不为“不确定字段”堆 fallback。
4. 展示字段没值时，优先不展示或展示“暂无数据”这类中性文案，不伪造业务内容。
5. 同一业务对象的状态尽量收敛到同一个 object，减少顶层平铺状态。

## 先读什么

1. `AGENTS.md`
2. `bd-fe-conventions`
3. 目标页面 / 组件
4. 同目录样式文件
5. 如果有 `docs/modules/<feature>.md`，先读需求卡
6. 如果目标文件依赖本地子组件或 helper，再按需读相关文件

## 输入要求

至少提供：

- 页面路径或组件路径

建议补充：

- 希望重点优化哪一部分，例如“接口状态”“模板层级”“data 状态”“课程模块”“弹窗逻辑”
- 是否允许顺手清理当前文件直接引用的本地组件

## 强规则

### 1. 功能不变

- 不改变页面主流程、跳转链路、接口参数、接口返回消费语义，除非用户明确要求。
- 不把已有可用功能“优化掉”。
- 如果某段逻辑看起来冗余，但无法确认是否被其他交互依赖，先保守处理。

### 2. 简化优先

- 能直接读接口对象的，不额外包 `resolveXxx`、`mapXxx`、`normalizeXxx`。
- 单页独占逻辑优先留在当前页，不为了“抽象统一”继续拆公共工具。
- 计算属性、方法、生命周期只保留当前页真正使用的部分。

### 3. 删除无依据兼容

- 不要为没有真实样例支持的字段名堆兼容分支。
- 不要写这类逻辑：`a.xx || a.xxx || a.data?.xx || '业务文案'`，除非这些分支都有真实依据。
- 如果字段在当前联调文档里已明确，就只按已明确字段消费。

### 4. 展示空值要克制

- 展示字段没值时，优先：
  - 不展示对应模块
  - 或展示 `暂无数据`、`暂无`、`待补充` 这类中性文案
- 不要自己补完整业务句子。
- 如果空值会误导用户，就直接隐藏该块。

### 5. 状态收敛

- 同一接口返回的数据，尽量按业务对象存储，例如：
  - `reportPayload`
  - `pageState`
  - `selectedItem`
  - `currentStep`
- 不要把同一对象拆成很多顶层字段平铺，造成同步困难。
- 需要模板简化时，优先用 `computed` 或项目等价派生机制从对象派生，不复制状态。

## 推荐优化方向

### 模板

- 删除明显重复、嵌套过深但不增加语义的结构。
- 相同卡片结构优先收敛成数据驱动，而不是重复写模板。
- 条件展示直接围绕“是否有值 / 是否需要展示”组织，不堆多层条件判断。

### 脚本

- 清理未使用的 `import`、状态、派生字段、方法、常量。
- 合并同类逻辑，例如多个仅改文案的小方法可并回派生逻辑。
- 请求流程保持清晰：请求前、成功、失败、结束四段明确，不混写。
- 与当前模式强相关的数据，优先由少量派生逻辑统一分流，而不是到处写 `if/else`。

### 样式

- 删除当前文件中已无模板引用的 class。
- 不因为“代码优化”顺手重做样式体系。
- 样式只做必要清理和小范围收口，保证页面视觉不跑偏。

## 执行步骤

1. 先确认目标路径和本次优化边界。
2. 通读页面，标出：
   - 真实业务对象
   - 主要请求流程
   - 页面状态分支
   - 明显冗余兼容
   - 明显误导性兜底文案
3. 识别哪些状态可以合并进同一个对象，哪些模板可以改成数据驱动。
4. 删除无依据兼容逻辑，只保留真实字段。
5. 清理空值展示：该隐藏就隐藏，需要保底时只用中性文案。
6. 清理无用代码：
   - 未使用 import
   - 未使用方法
   - 未使用派生逻辑
   - 未使用样式选择器
7. 回查功能路径，确保优化后：
   - 接口还能请求
   - 状态页还能显示
   - 点击事件还在
   - 跳转路径没变
8. 输出时说明本次收敛了哪些状态、删了哪些兼容、哪些风险没有动。

## 停止条件

遇到以下情况先停，不要硬改：

- 某段兼容逻辑虽然丑，但可能服务多个真实后端版本，且当前仓库没有依据判断
- 某个字段虽然没文档，但页面多个地方依赖，删掉可能影响线上
- 某个方法同时服务多个功能块，无法安全拆分
- 想继续优化必须改公共组件 / 公共工具

## 补充规则

1. 接口请求仍然复用项目现有请求入口，例如 `<project_request_entry-src/utils/requests.ts>` 或等价封装。
2. 涉及登录态仍然复用项目现有认证入口，例如 `<project_auth_entry-@/utils/auth.js>` 或等价封装。
3. 不顺手新增第二套状态管理或公共 helper。
4. 优先延续目标文件当前使用的组件写法和页面组织方式，不强行切换到另一套框架范式，例如 `<frontend_framework-Vue>` 的 Options API / Composition API，或等价模式之间的迁移。
5. 注释只保留高价值说明，解释“为什么这样收敛/删除”，不要写低价值废话。
6. 如果一个区块没有值就隐藏，模板上要让结构自然收口，不要留明显空白。
7. 如果参考页可复用，优先学习其组织方式，但不要机械复制无关逻辑。

## 标准触发模板

`使用 module-page-optimize-template，处理 [页面路径]。要求：不影响功能，简化代码流程，删除不必要兼容逻辑，收敛状态对象，展示字段没值时不要伪造文案。`

如果目标是组件：

`使用 module-page-optimize-template，处理 [组件路径]。重点优化模板层级、props 消费和空值展示，不改组件对外行为。`

## 输出要求

- 先说明本次优化边界。
- 明确列出主要收敛点：
  - 删除了哪些兼容
  - 合并了哪些状态
  - 隐藏了哪些空值展示
  - 保留了哪些风险逻辑未动
- 如果没法安全继续优化，要明确卡点，而不是继续猜。
