# 接口参数变更

## 适用时机

- 页面已经接过真实接口，当前只需要跟随后端调整请求参数。
- 变更范围集中在请求入参，例如字段名、字段层级、默认值、是否必传、数组转字符串、query/body 切换。
- 用户明确要求“只改参数”，或联调反馈确认问题只在请求侧。

## 目标与边界

- 目标是按真实接口契约调整请求参数。
- 只修改请求参数相关代码，不顺手修改返回解析、页面 UI、导航或业务流程。

## 先读什么

1. 优先读取 `docs/modules/<feature>.md`；如果没有，就根据用户描述定位目标页面和接口。
2. 读取 `AGENTS.md`、`bd-fe-conventions`、`<project_request_file-utils/request.js>`。
3. 查看目标页面或对应 `<project_utils_dir-utils/>` 里的请求调用点，确认当前参数来源和组装方式。
4. 如果用户给了最新接口文档、抓包或后端示例，严格以真实样例为准。

## 执行步骤

1. 先确认这次变更只影响请求参数，不包含返回结构、接口路径或页面交互改造。
2. 定位实际发请求的位置，只修改 `data`、`params`、路径拼接参数或调用前的参数组装逻辑。
3. 后端字段已经明确时，直接按真实字段名和结构改，不额外抽 `<param_mapper_name-mapParams>`、`<payload_builder_name-buildPayload>` 一类中间层。
4. 如果多个页面复用同一段参数构造，再考虑抽到当前模块下的复用函数；单页变更优先就地修改。
5. 改完后检查默认值、空值分支、类型转换和必传条件，避免把旧字段残留到请求里。

## 硬规则

- 只改请求参数相关代码，不顺手修改返回字段解析。
- 只改必要范围，不顺手调整页面 UI、文案、导航或交互。
- 所有请求继续走 `<project_request_file-@/utils/request.js>`，不要新增第二套请求封装。
- 没有真实接口字段说明时，不要猜参数名、嵌套结构或枚举值。
- 如果后端同时改了接口路径、返回结构或鉴权方式，明确提示应结合 `module-api` 一起处理。
- 如果旧参数已经废弃，要同步移除，避免新旧字段并存造成联调歧义。
- 需要添加注释时，只解释这次参数变更原因或兼容背景，不写低信息量注释。

## 不负责的内容

- 返回数据字段适配
- 页面 loading / empty / error 态补齐
- 新页面搭建、路由注册、视觉样式调整
- 无真实样例前提下的字段猜测和接口兜底设计

## 收尾建议

- 说明本轮改了哪些请求调用点、参数字段和默认值分支。
- 如果存在接口契约缺口、字段歧义或返回结构变更，明确列为后续待处理项。
