# Codex Agent Playbook

## Purpose

Use this playbook after `TSTypeConversion` has already produced a `--suggest-json` package.

The converter is responsible for:

- generating TypeScript types
- deciding target file paths
- returning structured modification suggestions

The agent is responsible for:

- reading the target file
- deciding how to merge generated code into existing code
- applying the patch
- verifying the result

## Recommended execution flow

1. Run the converter with `--suggest-json`
2. Read the returned package
3. For each `item`:
   - open `file`
   - inspect existing declarations
   - use `symbols` as the generated declaration scope
   - use `content` as the generated truth
   - preserve unrelated manual code
4. Apply a patch
5. Re-read the file and verify the expected symbols exist

## What the agent should treat as authoritative

- `file`
  - authoritative target path
- `content`
  - authoritative generated TypeScript body
- `symbols`
  - authoritative generated symbol set
- `source_paths`
  - traceability only
- `operation_ids`
  - traceability only
- `merge_hint`
  - guidance, not a hard rule

## Minimal decision rules

### Case 1: target file does not exist

Do:

- create the file with `content`

### Case 2: target file exists and generated symbols do not exist

Do:

- append `content`
- keep unrelated manual code unchanged

### Case 3: target file exists and generated symbols already exist

Do:

- inspect the existing declarations first
- patch only the matching generated declarations if the intent is obvious
- preserve unrelated declarations

Do not:

- blindly overwrite the whole file
- remove unrelated exports

### Case 4: the file structure is ambiguous

Do:

- stop and surface the ambiguity
- prefer a targeted follow-up edit over a risky bulk rewrite

## Conflict heuristics

Treat these as warning signs:

- the same symbol exists with different semantics
- the target file contains dense manual organization or hand-maintained exports
- imports would need major reorganization
- the generated `content` would clearly duplicate existing declarations

In these cases, prefer:

- smaller patch scope
- explicit review
- or a follow-up pass

## Patch strategy

Preferred order:

1. replace matching generated declarations if they are easy to isolate
2. append missing generated declarations
3. only adjust imports if required by the generated content

Avoid:

- formatting the whole file unless necessary
- moving unrelated declarations
- reordering the file aggressively

## Post-edit verification

After patching, verify at minimum:

- the file still parses visually
- all expected `symbols` are present
- obvious duplicates were not introduced
- the generated block matches the intent of `content`

If the repository has a fast verification command, run it.

## Suggested Codex prompt template

Use this template after obtaining the suggestion package:

```text
You are applying TypeScript code generated by TSTypeConversion.

Rules:
- Treat `file` as the target file path.
- Treat `content` as the generated source of truth for the listed `symbols`.
- Preserve unrelated manual code.
- Do not overwrite the whole file unless the file does not exist.
- Read the target file before editing.
- If matching symbols already exist, patch them carefully instead of blindly appending.
- If the merge is ambiguous, stop and explain the ambiguity.

Task:
1. Read the suggestion item.
2. Open the target file.
3. Merge the generated declarations into the file.
4. Verify the expected symbols exist after the edit.
5. Summarize what changed.
```

## Suggested human workflow

1. Run `--suggest-json`
2. Hand the JSON to Codex agent
3. Let the agent patch target files
4. Review the diff
5. Run project checks if needed

This keeps the converter focused on deterministic generation and lets the agent handle project-specific merge decisions.
