# 多项目 skill 规则传递设计

## 1. 背景

`TSTypeConversion` 作为一个通用 `skill`，会被复制到不同项目中使用。

但不同项目的类型输出规则往往不同，例如：

- 有的项目希望输出到 `src/api/user/types.ts`
- 有的项目希望输出到 `src/modules/user/model.ts`
- 有的项目按 `tag` 分模块
- 有的项目按 `path` 分模块
- 有的项目会把请求和响应拆到不同文件

因此：

**规则不能写死在 skill 内部。**

正确做法是：

- `skill` 提供“读取规则并执行”的能力
- 每个项目提供自己的配置文件
- agent 在调用 skill 时，把项目规则传进去，或让 skill 自动发现

## 2. 核心原则

### 2.1 规则属于项目，不属于 skill

`skill` 是通用能力。

项目规则是项目自己的工程约束。

所以两者职责要分开：

- `skill`
  - 定义怎么读取规则
  - 定义规则格式
  - 定义执行优先级
- 项目
  - 提供具体规则文件
  - 决定输出路径和模块归档方式

### 2.2 规则传递优先级要固定

为了避免同一项目在不同环境下行为不一致，必须定义稳定优先级。

推荐优先级：

1. CLI 显式传入 `--output-config`
2. 环境变量 `TS_TYPE_CONVERSION_CONFIG`
3. 自动发现项目根目录默认配置文件
4. 都没有时退回默认单文件输出

### 2.3 大模型不直接携带“隐式规则”

大模型可以读取项目代码并推荐规则，但不能把规则只存在“会话上下文”里。

因为这样会导致：

- 规则不可追踪
- 不可复用
- 不可测试
- 下次运行无法复现

所以最终规则必须落成项目文件。

## 3. 推荐的规则传递方式

推荐同时支持三种方式。

### 3.1 方式一：CLI 显式传入配置文件

适合：

- CI
- 命令行手工执行
- agent 已经知道配置文件路径

示例：

```bash
python -m scripts.cli \
  --input http://127.0.0.1:9000/swagger.json \
  --style interface \
  --output-config ./openapi-ts.config.yaml
```

优点：

- 最明确
- 最稳定
- 最适合自动化

### 3.2 方式二：环境变量传递

适合：

- IDE 集成
- agent 框架统一注入环境变量
- CI/CD 场景

示例：

```bash
export TS_TYPE_CONVERSION_CONFIG=./openapi-ts.config.yaml
python -m scripts.cli --input ./swagger.json --style interface
```

优点：

- 易于接入自动化环境
- 命令更短

缺点：

- 不如 CLI 显式参数直观

### 3.3 方式三：项目内自动发现

适合：

- skill 被复制到不同项目中时的默认使用方式
- 用户不想每次都传配置路径

推荐自动发现的文件名：

- `openapi-ts.config.yaml`
- `openapi-ts.config.yml`
- `ts-type-conversion.config.yaml`
- `ts-type-conversion.config.yml`

推荐查找位置：

1. 当前工作目录
2. 当前工作目录向上查找到项目根目录
3. skill 根目录下的示例配置仅作为参考，不作为项目规则

优点：

- 使用体验最好
- 适合“复制 skill 到项目后直接可用”

## 4. 推荐的优先级顺序

建议最终实现时按以下顺序解析规则来源：

### 4.1 第一步：看 CLI 参数

如果用户明确传了：

```bash
--output-config ./openapi-ts.config.yaml
```

则直接使用该文件。

### 4.2 第二步：看环境变量

如果没有 CLI 参数，则读取：

```bash
TS_TYPE_CONVERSION_CONFIG
```

### 4.3 第三步：自动发现配置文件

如果以上都没有，再自动查找：

- `openapi-ts.config.yaml`
- `ts-type-conversion.config.yaml`

### 4.4 第四步：无配置时退回默认模式

如果仍未找到配置：

- 使用 `--output` 输出单一文件
- 或提示用户当前未配置项目级输出规则

## 5. 推荐的配置文件格式

建议用 YAML。

原因：

- 可读性好
- 适合人工维护
- 适合 agent 生成初版
- 比 JSON 更适合写多条规则

推荐配置示例：

```yaml
version: 1

default_output: src/api/common/types.ts

rules:
  - match:
      tags: ["user"]
    output: src/api/user/types.ts

  - match:
      tags: ["auth"]
    output: src/api/auth/types.ts

  - match:
      path_prefix: /admin
    output: src/api/admin/types.ts

  - match:
      operation_ids: ["login", "refreshToken"]
    output: src/api/auth/types.ts
```

## 6. 配置字段建议

### 6.1 顶层字段

建议支持：

- `version`
- `default_output`
- `rules`

### 6.2 `rules` 字段

每条规则建议包含：

- `match`
- `output`

后续可扩展：

- `mode`
- `append`
- `group_by`

### 6.3 `match` 字段

建议首期支持这些匹配条件：

- `tags`
- `path_prefix`
- `operation_ids`

后续可以扩展：

- `methods`
- `paths`
- `style`

## 7. agent 如何把规则传给 skill

这里是最关键的一点。

### 7.1 最推荐的方式

agent 不要把规则“写在 prompt 里长期携带”。

而应该做成：

1. agent 先在项目里找到配置文件
2. agent 调用 skill 对应的 CLI
3. 通过 `--output-config` 把规则路径传给脚本

也就是：

```bash
python -m scripts.cli \
  --input ./swagger.json \
  --style interface \
  --output-config ./openapi-ts.config.yaml
```

这样最清晰。

### 7.2 如果 agent 是第一次接入某个项目

可以这样处理：

1. agent 读取项目目录结构
2. agent 读取已有 `api/`, `services/`, `models/`, `types/` 目录
3. agent 生成一份推荐配置
4. 用户确认
5. 保存为项目根目录配置文件
6. 后续所有执行都读这份文件

这里大模型的作用是：

- 帮你建立规则

不是：

- 每次执行时都重新发明规则

## 8. skill 内应该怎么描述这件事

在 `SKILL.md` 中，建议明确写清楚：

- 这个 skill 支持项目级配置文件
- 优先读取 `--output-config`
- 若未传入，则尝试自动发现项目配置
- 若仍未发现，则退回单文件输出

也就是说，skill 本身不保存项目规则，只消费项目规则。

## 9. 不同项目的典型使用方式

### 9.1 项目 A：前后端分离，按模块输出

配置可能是：

```yaml
default_output: src/api/common/types.ts

rules:
  - match:
      tags: ["user"]
    output: src/api/user/types.ts

  - match:
      tags: ["auth"]
    output: src/api/auth/types.ts
```

### 9.2 项目 B：按业务目录组织

配置可能是：

```yaml
default_output: src/shared/api/types.ts

rules:
  - match:
      path_prefix: /user
    output: src/modules/user/api.types.ts

  - match:
      path_prefix: /order
    output: src/modules/order/api.types.ts
```

### 9.3 项目 C：一个 swagger 输出一个文件

这种项目甚至可以不写规则文件。

直接：

```bash
python -m scripts.cli \
  --input ./swagger.json \
  --style interface \
  --output src/types/generated-api.ts
```

## 10. 建议的后续实现点

建议后续给项目补以下能力：

### 10.1 CLI

新增：

- `--output-config`

行为：

- 读取项目规则配置文件
- 决定接口输出到哪些文件

### 10.2 自动发现

支持自动查找项目配置文件。

### 10.3 校验

在执行前校验配置是否合法，例如：

- 路径字段是否为空
- `rules` 格式是否正确
- 多条规则是否冲突

### 10.4 解释模式

建议未来增加一个调试参数，例如：

- `--explain-routing`

输出示例：

- `/user/login -> src/api/auth/types.ts`
- `/user/profile -> src/api/user/types.ts`

这样便于排查规则命中逻辑。

## 11. 最终建议

针对“不同项目规则不同，怎么传递给这个 skill”这个问题，最稳的答案是：

- 规则保存在项目自己的配置文件中
- skill 只负责读取和执行这些规则
- agent 通过 CLI 参数、环境变量或自动发现把规则交给 skill
- 大模型只负责首次推荐规则，不负责每次都临时决定规则

一句话总结：

**把规则做成项目配置文件，把 skill 做成规则消费者，把 agent 做成配置发现和调用入口。**
