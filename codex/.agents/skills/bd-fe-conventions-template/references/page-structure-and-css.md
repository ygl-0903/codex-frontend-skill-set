# 页面结构与 CSS 通用规则

适用范围：项目内页面开发、设计稿回填、样式整理。  
目标：统一页面拆分方式、`className` 命名、样式组织和 CSS 书写边界。

## 1. 布局划分

页面优先按职责拆成三层，不按视觉细枝末节先拆。

- 页面层：负责整体布局、留白、分栏、滚动区、头尾结构。
- 模块层：负责可独立理解的业务区块，如筛选区、表格区、表单区、统计区、弹窗主体。
- 元素层：只服务于某个模块的标题、按钮区、说明文案、数值、空态等局部内容。

判断原则：

- 能单独理解、可能复用、可能拆成子组件的，定义为模块。
- 离开所属模块就失去语义的，定义为元素。
- 纯装饰包裹层优先挂到现有模块或元素上，不单独起业务名。
- 一个容器不要同时承担页面布局和具体业务语义；冲突时拆成两层。

## 2. `className` 定义

默认采用语义化命名，优先 `BEM` 风格。

- block：`module-name`
- element：`module-name__element-name`
- modifier：`module-name--modifier`
- state：`is-active`、`is-disabled`、`is-loading`

命名要求：

- 只用小写英文和中划线。
- 优先表达角色和语义，不表达颜色、位置、序号或临时视觉特征。
- 同一语义尽量全页只保留一个词。
- modifier 表示稳定变体，state 表示即时状态，不混写。

禁止项：

- 不用 `id` 作为主选择器。
- 不用 `left-box`、`blue-card`、`item2` 这类弱语义名字。
- 不把第三方组件内部类名当作业务命名。

## 3. 样式组织方式

样式写法跟随项目现有方案，不另起一套。

- 项目用 `less` / `scss`：优先使用嵌套、变量、mixin、分段组织，不要全部平铺成一长串普通 CSS。
- 项目用 `styled-components`：按组件维度共置样式和状态，不回退到大段全局样式思维。
- 项目用 `CSS Modules`：按模块作用域组织，不绕开局部作用域写全局污染样式。

通用要求：

- 先遵守项目既有方案和目录结构，再写具体样式。
- 同一文件建议按“页面层 -> 模块层 -> 元素层 -> modifier/state -> 第三方覆盖”组织。
- 复用能力只在确实跨页面或跨模块时再抽，不为一两个节点提前抽象。

## 4. CSS 书写规范

- 优先低复杂度选择器：单类选择器优先，必要时再用 block + element 或状态组合。
- 避免深层后代选择器、标签选择器叠加、为压优先级不断堆层级。
- 样式默认局部收敛到当前页面或当前模块，避免无宿主的全局覆盖。
- 覆盖第三方 UI 组件时，先挂在自己的宿主模块下，再做局部覆盖。
- 状态样式与结构样式分开写，避免把布局、视觉、业务状态混成一个类。
- 同类间距、字号、圆角、边框、颜色优先复用现有 token、变量或公共约定，避免大量魔法数字和重复硬编码。
- 能靠结构调整解决的问题，不优先靠补丁式 CSS 兜底。

## 5. 最小检查清单

开始写页面前至少确认：

1. 页面层、模块层、元素层是否已分清。
2. `className` 是否语义化、稳定、可维护。
3. 样式是否按项目当前 `less/scss/styled-components/CSS Modules` 方案组织，而不是全部平铺。
4. 选择器和第三方覆盖是否控制在最小作用域内。
5. 是否存在明显可复用却未复用、或明显不该抽却被提前抽象的样式。

## 6. 参考原则

- BEM：命名清晰、块与元素边界明确。
- SMACSS：布局与模块职责分离。
- MDN：控制选择器复杂度，保持样式可读和可维护。
